- Correct: Generic AI SDK tool detection uses `isToolUIPart`/`getToolName` from `ai` in `src/components/chat-interface.tsx:86-109`, which is aligned with AI SDK v6 static/dynamic tool part helpers.
- Correct: The UI now has a rendering path for non-artifact tool parts instead of silently returning `null` (`src/components/chat-interface.tsx:569-644`). This directly addresses the H2O hidden `tool-loadSkill` visibility gap.
- Correct: Reconciliation telemetry is defensive: `safeEmitTelemetry` catches callback errors so logging cannot break chat recovery (`src/lib/chat-reconciliation.ts:44-52`).
- Correct: Browser telemetry payloads avoid message text, file URLs, attachment names, tokens, and tool outputs. They log thread/message IDs, roles, counts, tool state decisions, and error messages only (`src/components/chat-interface.tsx:297-306`, `src/components/chat-interface.tsx:359-365`, `src/components/chat-interface.tsx:377-392`).
- Correct: Dirty diff includes focused tests for helper contracts and reconciliation decision telemetry (`src/components/chat-interface.test.tsx:160-326`). Per task context, focused tests and `bunx tsc --noEmit` passed after apply. I also ran `git diff --check`; it produced no whitespace errors.

- Blocker: Terminal tool cards will not reliably open when a live tool transitions from in-flight to terminal. `Tool` is an uncontrolled Radix/base collapsible wrapper that forwards `defaultOpen` only as the initial state (`src/components/ai-elements/tool.tsx:20-26`). The changed render passes `defaultOpen={toolMetadata.defaultOpen}` (`src/components/chat-interface.tsx:580-583`, `src/components/chat-interface.tsx:615-618`), but if the same keyed tool was first mounted with `input-available`/`input-streaming`, later `output-available`/`output-error` changes will not reopen it. Recommendation: make terminal auto-open controlled or force a remount only when the terminal-open boundary changes, while preserving user toggling where intended. Add a component lifecycle test, not just a pure helper test.
- Blocker: Reconciliation uses `messagesRef.current` as the live-state guard, but `onFinish` receives the authoritative finished messages and does not copy them into the ref before starting reconciliation (`src/components/chat-interface.tsx:297-307`, `src/components/chat-interface.tsx:350-352`, `src/components/chat-interface.tsx:373-376`). If React subscription/effect state lags the AI SDK `onFinish` callback, `getCurrentMessages()` can be stale and the new stale-overwrite guard in `persistedSnapshotDecisionReason` (`src/lib/chat-reconciliation.ts:55-77`) may incorrectly allow an older persisted snapshot. Recommendation: set `messagesRef.current = finishedMessages` inside `onFinish` before logging/reconciliation, or pass an explicit current-message snapshot into `reconcileThreadAfterStream`. Add a regression test where the ref/current messages lag the `onFinish` payload.

- Note: `asToolPart` accepts `unknown` but calls `isToolUIPart(candidate)` without first proving `candidate.type` is a string (`src/components/chat-interface.tsx:86-89`). AI SDK's `isToolUIPart` calls `part.type.startsWith(...)`, so `getToolRenderingMetadata({})` can throw despite its `unknown` signature. Current render paths pass real message parts, so this is not a current production blocker. Recommendation: guard `typeof candidate?.type === "string"` before calling `isToolUIPart`, or narrow the helper input type.
- Note: Reconciliation telemetry logs one `attempt` event plus one decision event per poll (`src/lib/chat-reconciliation.ts:136-149`), up to 12 attempts by default (`src/lib/chat-reconciliation.ts:82-87`). That is acceptable for temporary browser diagnostics, but noisy for long-term production. Recommendation: gate under a debug flag or reduce to summary logging after the H2O repro is diagnosed.
- Note: `console.warn` logs raw error messages in the browser (`src/components/chat-interface.tsx:377-385`; `src/lib/chat-reconciliation.ts:167-174`). This is usually fine for client diagnostics, but can expose backend/internal details if server action errors include them. Recommendation: sanitize/classify errors if these logs remain beyond the diagnostic slice.
- Note: The reconciliation polling window is now ~24.5 seconds by default (`settleMs=500`, `maxAttempts=12`, `retryMs=2000`; `src/lib/chat-reconciliation.ts:82-88`). This is a behavior change from one fetch to polling and is directionally useful, but still much shorter than the known ~240s H2O timeout window. Not a blocker for telemetry/tool visibility, but it should not be considered a full fix for late persistence.
